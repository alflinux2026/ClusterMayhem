# ✅ 1. ESTADO GENERAL

El cluster ha alcanzado un estado estable con:

- Leader election determinista funcionando
- Heartbeat-based cluster state sincronizado
- Nodos operativos sin lógica de consenso compleja
- Sistema coherente entre 3 nodos

---

# 🧱 2. ARQUITECTURA ACTUAL

## 🔹 Cluster Layer (global)

- `cluster_state` = fuente de verdad
- `compute_leader()` = función pura global

Regla:

```text
leader = nodo con menor priority entre nodos vivos
````

---

## 🔹 Node Layer (local)

Cada nodo:

* mantiene estado local (BOOT / STANDBY / ACTIVE)
* envía heartbeat
* no toma decisiones de liderazgo

---

# 🔁 3. FLUJO ACTUAL

```text
heartbeat → cluster_state → compute_leader → state transition
```

---

# ⚙️ 4. COMPONENTES CLAVE

## NodeRuntime

* tick() → BOOT → STANDBY
* refleja estado del leader
* no contiene lógica de election

## Leader

* función pura
* depende solo de cluster_state

## Heartbeat

* actualiza cluster_state
* incluye: node_id, priority, last_seen

---

# 🚫 5. LO QUE YA SE ELIMINÓ

* ElectionEngine
* try_become_leader()
* leader decisions dentro del nodo
* prioridad dinámica
* consenso distribuido

---

# 🧠 6. PRINCIPIOS DEL SISTEMA

```text
Nodes no deciden.
Cluster decide.
Leader es función, no estado.
```

---

# ⚠️ 7. LIMITACIONES CONOCIDAS

* Sin quorum (no Raft)
* No hay persistencia de eventos
* No hay anti-flapping del leader aún
* Estados ACTIVE/STANDBY solo reflejan ejecución local

---

# 🚀 8. SIGUIENTE FASE (ROADMAP)

## Fase 2 — Event Routing Layer

Objetivo:

* cliente envía eventos a cualquier nodo
* leader actúa como router
* nodos ejecutan eventos
* flujo completo de datos end-to-end

---

# 🧭 9. OBJETIVO FINAL DEL SISTEMA

```text
Cliente → Nodo → Leader → Nodo → Ejecución
```

Cluster como sistema distribuido funcional, no solo election layer.
