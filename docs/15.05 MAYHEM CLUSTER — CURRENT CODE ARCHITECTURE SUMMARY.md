# MAYHEM CLUSTER — CURRENT CODE ARCHITECTURE SUMMARY

Date: 2026-05-15
Status: PRE-CONSOLIDATION
Phase: CONTROL PLANE STABILIZATION

---

# 1. CURRENT PROJECT GOAL

Mayhem Cluster is currently evolving from:

```text
distributed heartbeat demo
```

into:

```text
deterministic distributed cluster runtime
```

The system already supports:

* distributed node discovery
* heartbeat propagation
* node expiration
* leader election
* failover
* cluster convergence

Current focus is stabilizing the CONTROL PLANE before implementing WAL
and distributed DATA PLANE features.

---

# 2. CURRENT HIGH-LEVEL ARCHITECTURE

Current architecture is composed of two partially overlapping models:

## OLD MODEL (simulation/local-state model)

Based on:

* LeaseManager
* local leases
* local election logic
* HeartbeatWorker

Originally designed for in-memory simulation.

---

## NEW MODEL (real distributed model)

Based on:

* FastAPI networking
* HTTP heartbeats
* distributed cluster_state
* TTL cleanup
* distributed convergence

This is now the dominant architecture and the intended future direction.

# 4. CURRENT COMPONENT BREAKDOWN

# CONFIG LAYER

## config/nodes.json

Defines static cluster topology.

Contains:

* node_id
* host
* port
* priority

Example:

```json
{
  "node_id": "lnx200nas",
  "host": "100.100.1.200",
  "port": 7000,
  "priority": 1
}
```

---

## config/loader.py

Simple helper to load node configuration.

---

# STATE MODEL

## runtime/state.py

Defines node lifecycle states:

```text
BOOT
DISCOVERING
STANDBY
ACTIVE
DEGRADED
ISOLATED
OFFLINE
```

This is intended to become the canonical runtime state machine.

---

## core/transitions.py

Defines valid state transitions.

Example:

```text
BOOT -> DISCOVERING
DISCOVERING -> ACTIVE
ACTIVE -> DEGRADED
```

Provides:

```python
can_transition(from_state, to_state)
```

---

## core/node.py

Simple stateful node object using transition validation.

Currently partially disconnected from runtime implementation.

---

# RUNTIME LAYER

## node/node_runtime.py

Main runtime object representing a cluster node.

Responsibilities:

* node identity
* node state
* leadership behavior
* heartbeat emission
* lease interaction
* runtime ticking

Important methods:

```python
tick()
try_become_leader()
emit_heartbeat()
transition()
```

This is currently one of the central runtime components.

---

## runtime/node_boot.py

Current runtime bootstrap file.

Responsibilities currently mixed together:

* FastAPI app
* API endpoints
* worker startup
* node bootstrap
* cluster mutation
* leader calculation
* runtime lifecycle

This file is currently overloaded and scheduled for refactor.

---

## runtime/cluster_store.py

Current distributed shared memory model.

Defines:

```python
cluster_state = {}
```

and:

* TTL cleanup
* active cluster view
* node expiration

Important functions:

```python
cleanup_cluster()
get_active_cluster()
```

This is becoming the future single source of truth.

---

## runtime/cluster_view.py

Currently minimal placeholder object.

Not yet fully integrated.

Potential future purpose:

* consistent cluster snapshots
* derived views
* leader visibility
* topology abstraction

---

# NETWORKING LAYER

## transport/server.py

Main FastAPI transport server.

Responsibilities:

* receive heartbeats
* expose /cluster
* expose /leader
* update cluster_state
* cleanup dead nodes

Endpoints:

```text
GET  /health
GET  /cluster
GET  /leader
POST /heartbeat
```

This is becoming the canonical cluster mutation layer.

---

## transport/client.py

Outgoing networking layer.

Responsibilities:

* send heartbeat
* broadcast heartbeat

Functions:

```python
send_heartbeat()
broadcast_heartbeat()
```

Uses HTTP requests between peers.

---

# WORKER LAYER

## workers/cluster_worker.py

Main distributed worker loop.

Current behavior:

```python
register_local_node()
emit_heartbeat()
sleep()
```

Runs continuously.

This worker is part of the new distributed architecture.

---

## workers/heartbeat_worker.py

Legacy simulation-oriented worker.

Responsibilities:

* lease renewals
* lease expiration
* local heartbeat simulation

Part of the older local-memory architecture.

Scheduled for removal.

---

# LEADER ELECTION

## election/election_engine.py

Current simplistic election logic.

Current rule:

```python
if active leader exists:
    cannot become leader
else:
    may become leader
```

Still dependent on LeaseManager.

---

## election/leader.py

Simple deterministic selection:

```python
min(priority)
```

Lower priority number wins.

Currently not fully integrated with distributed cluster_state.

---

# LEASE SYSTEM

## lease/lease.py

Lease model with TTL expiration.

---

## lease/lease_manager.py

Current local lease tracking system.

Responsibilities:

* grant leases
* expire leases
* validate leases
* track active leader

This was useful for local simulation but now conflicts with distributed
cluster_state architecture.

Scheduled for removal.

# CURRENT LEADER FLOW

Leader is currently computed using:

```python
min(priority)
```

among nodes considered alive.

However:

* logic exists in multiple places
* BOOT nodes may incorrectly participate
* LeaseManager still interferes

This area is currently transitional.

# CURRENT MAJOR PROBLEMS

# 1. DOUBLE SOURCE OF TRUTH

Currently both exist:

```text
LeaseManager
cluster_state
```

This creates risk of divergence.

Planned solution:

```text
cluster_state only
```

---

# 2. DISTRIBUTED MUTATION SURFACE

Multiple components mutate cluster state:

* workers
* endpoints
* bootstrap code

This risks race conditions and non-determinism.

Planned solution:

```text
POST /heartbeat only
```

---

# 3. DUPLICATED LEADER LOGIC

compute_leader exists in multiple locations.

Planned solution:

```text
runtime/leader.py
```

single implementation.

---

# 4. LOCAL PRIVILEGED WRITES

Current workers directly mutate local cluster_state.

This bypasses distributed networking semantics.

Planned solution:

```text
self-heartbeat via HTTP
```

Self must behave like any other peer.

---

# 5. NO THREAD SAFETY

cluster_state currently has no locking.

Potential issues:

* concurrent mutation
* inconsistent reads
* corruption

Planned solution:

```python
threading.Lock()
```

---

# 6. LEADER INSTABILITY

Current leader election has no temporal stability.

Possible result:

```text
leader bouncing
```

Planned solution:

```text
stability window
```

before leader promotion.

---

# 7. STATE SEMANTICS NOT FULLY CLEAN

Current temporary hack:

```python
BOOT == ACTIVE
```

This is unsafe for future WAL and replication logic.

Planned solution:

Only:

```text
ACTIVE
```

may lead.
