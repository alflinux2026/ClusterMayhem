# Arquitectura Actual

## Leader

Responsable de:

* detectar CREATED
* seleccionar worker
* cambiar estado -> EXECUTING
* persistir estado
* enviar al worker
* marcar COMPLETED

## Worker

Responsable únicamente de:

* ejecutar payload
* devolver resultado

El worker NO persiste estados finales.

# Bug Principal Descubierto

## Síntoma

El reconciler:

* recreaba CREATED
* dispatch reenviaba
* leader hacía WORKER SEND

PERO:

* worker aparentemente "no ejecutaba"

---

# Causa Real

El worker sí recibía el evento.

El problema estaba en:

```python
executed_keys
```

La idempotencia local del worker seguía viva en RAM aunque:

* leader hubiera reiniciado
* log hubiera perdido COMPLETED

Entonces:

```python
if execution_key in executed_keys:
    CACHE HIT
```

bloqueaba silenciosamente la reejecución.

---

# Descubrimiento Clave

El problema NO estaba en:

* reconciler
* dispatch
* forward_event
* HTTP
* routing

El problema era:

```text
worker local memory surviving leader replay
```

---

# Solución Temporal Aplicada

## Idempotency cache desactivada

Se comentó:

```python
if event.execution_key in executed_keys:
```

para permitir recovery real.

---

# Fix Importante Aplicado

En dispatch:

```python
event.attempt = (event.attempt or 0) + 1
```

antes de:

```python
event.mark_status(EXECUTING)
```

Esto permite:

* retries reales
* execution_key distinto por intento
* recovery funcional

# Conclusión

Hoy el cluster dejó de ser un experimento local y pasó a comportarse como un runtime distribuido coherente.

El principal avance no fue el reconciler.

Fue descubrir:

* dónde muere realmente una reejecución distribuida
* cómo interactúan replay + cache local + recovery
* y por qué el leader y el worker pueden divergir aunque el log sea correcto.

Eso desbloquea el diseño real de retries distribuidos para la siguiente fase.
