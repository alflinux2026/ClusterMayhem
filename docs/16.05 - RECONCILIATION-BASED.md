# 🧠 SNAPSHOT — Cambio de modelo de tareas del Leader (Mayhem Cluster)

## 📌 Estado actual (modelo antiguo)

El leader está actuando como un **orquestador secuencial tipo pipeline**, mezclando responsabilidades:

### 🔁 Flujo actual implícito
```

/event
→ compute_leader
→ route_event
→ append_event (executing)
→ forward worker
→ worker executes
→ ACK vuelve al leader
→ append_event (completed)

```

### ❌ Problemas del modelo actual
- El leader mezcla:
  - routing
  - scheduling
  - persistencia
  - control de estado
- El worker participa indirectamente en la verdad del sistema
- El log depende del flujo runtime
- Riesgo de:
  - estados huérfanos (executing sin completion)
  - pérdida de ACK si leader cae
  - inconsistencia de event_log
- No existe separación clara entre “decidir” y “ejecutar”

---

## 🎯 Nuevo modelo (RECONCILIATION-BASED)

El leader deja de ser un pipeline y pasa a ser un **conjunto de loops independientes sobre un event log único**.

---

## 🧾 Principio base

> El event_log es la única fuente de verdad (single source of truth)
> y todos los procesos operan sobre él.

---

## 🧩 3 loops independientes del sistema

### 1. 📥 RECEIVE LOOP (ingestion)
Responsabilidad: persistencia inicial

```

/event → append_event(status="created")

```

✔ Solo escribe en log
✔ No ejecuta
✔ No decide workers

---

### 2. 🚦 DISPATCH LOOP (scheduler)

Responsabilidad: convertir intención en ejecución

```

for event in log:
if status == "created":
assign_worker(event)
mark_status("executing")
append_event(event)

```

✔ Asigna trabajo
✔ Marca ejecución
✔ No ejecuta directamente

---

### 3. 🔁 ACK / RECONCILIATION LOOP (reaper)

Responsabilidad: cierre y recuperación

```

for event in log:
if status == "executing":
if worker_completed_signal:
mark_status("completed")
append_event(event)

```
    if timeout_expired:
        requeue(event)
```

```

✔ Detecta fallos
✔ Reintenta ejecuciones
✔ Finaliza estados colgados

---

## 🧠 Nuevo rol del leader

El leader deja de ser “ejecutor” y pasa a ser:

> 🧠 **Event State Machine Controller**

### Ya no hace:
- ejecución directa
- forward síncrono como parte del flujo crítico
- escritura dependiente de workers

### Ahora hace:
- lectura del log
- transición de estados
- scheduling
- reconciliación

---

## ⚙️ Workers (nuevo rol)

- Ejecutan tareas asignadas
- No escriben en el event_log
- No determinan estado global
- Son reemplazables y stateless

---

## 🧾 Modelo de estados del evento

```

created
↓
executing
↓
completed
↓
(optional: failed / retry)

```

✔ “executing” es estado no confiable (transitorio)

---

## 💥 Propiedades que se ganan

- ✔ Event log consistente (single writer lógico)
- ✔ Recuperación automática ante fallos del leader
- ✔ Workers desacoplados del sistema de verdad
- ✔ Reintentos seguros (at-least-once)
- ✔ Posibilidad de replay completo del sistema

---

## ⚠️ Cambio conceptual clave

### Antes:
> Leader = orquestador de ejecución

### Ahora:
> Leader = reconciler de estado basado en log

---

## 🚀 Resultado final

El sistema deja de ser un pipeline y pasa a ser:

> 🧩 **Event Sourcing + Reconciliation Loops + Stateless Workers**
